CAP theorem states that a distributed database system can only provide two of three guarantees:
 - Consistency (all nodes see the same data)
 - Availability (every request gets a response)
 - Partition Tolerance (system operates despite network failures).
Cassandra is by default an AP database.

##### Important Memory Note from Google Gemini #####
You are running 9 databases plus an application. This is extremely RAM-intensive. 
Ensure your Docker Desktop / Engine has at least 12GB–16GB of RAM allocated, or several containers
(especially Oracle and SQL Server) will fail to start.
Startup Time: With all these healthchecks, the first time you run docker-compose up,
it might take 2–3 minutes before the study28 container actually starts, as it will wait for the slowest database
(usually Oracle or Cassandra) to be fully ready.
Does your local machine have enough RAM to handle all these instances at once, or are you running them selectively?

From AI: added xpack.security.enabled=false. In modern Elasticsearch (8.x and 9.x), security is on by default. To keep it consistent with your other repositories where you are likely connecting without complex SSL/Auth setups for local study, disabling it is usually the path of least resistance.
  A few tips for Elasticsearch 9.3.0:
  Virtual Memory: On Windows/Docker Desktop, this usually works fine. However, if the container crashes immediately with an "exit code 137" or a "max virtual memory areas" error, you might need to run wsl -d docker-desktop and execute sysctl -w vm.max_map_count=262144.
  Healthcheck: I included a healthcheck using curl. This ensures your Node.js app doesn't try to connect before the Elasticsearch engine has fully finished its internal "bootstrap" checks.
  Are you planning to use the official @elastic/elasticsearch client for the repository implementation in your Node.js app?
